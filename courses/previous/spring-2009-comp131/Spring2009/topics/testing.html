COMP 131 Testing Rules

General Testing Rules

  • always give a test method a descriptive name that starts with the word "test"
  • always record test cases for each constructor or method immediately after writing it

Testing Constructors and Accessors

  1. use one test method for each constructor
  2. create an object using the constructor
  3. call an accessor method to check the value of each field

Note that you do not need to create a separate test method for each accessor.

Testing Mutators

  1. use at least one test method for each mutator
  2. create an object using a constructor
  3. call the mutator
  4. call an accessor method to check the value of the field that was changed

Testing Methods that Contain if statements

  1. write sufficient tests to achieve statement coverage
  2. use a different test method for each different segment of code that can be executed when the method is called.

    Examples:

    • the following method: 
      public int spindle(int x) {
    if (x < 3) {
        return x + 1;
    } else {
        return x + 2;
    }
}
      requires at least two test methods: one for x < 3 and one for x >= 3
    • the following method: 
      public int mutilate(int x, int y) {
    if (x < 3) {
        if (y > 4) {
            return y;
        } else {
            return x + 2;
        }
    } else if (y < 3) {
        return y + 1;
    } else {
        return x - y;
    }
}
      requires at least four test methods:
      • one for x < 3 && y > 4
      • one for x < 3 && y <= 4
      • one for x >= 3 && y < 3
      • one for x >= 3 && y >= 3

Testing Methods that Only Print

For methods that only print to the terminal window (no return value or change of field values):
  1. call the method
  2. manually inspect the output in the terminal window
It is a good idea to clear the terminal window before running your tests when you have this kind of test cases.

Testing Constructors that Create a Collection

  1. create a test method for the constructor (see the rule for Constructors above)
  2. in that test method, check the size of the new collection to ensure that the ArrayList or the Array was created correctly by the constructor.

Testing Methods that Add to a Collection

  1. add several elements to the collection
  2. check the size of the collection
  3. get the first element of the collection, and check enough fields to be sure it is the one you expect
  4. get the last element of the collection, and check enough fields to be sure it is the one you expect

    Note that you must also have a method that gets an element from the collection (by index).

Testing Methods that Get an Element from a Collection by Index

  1. create a test method to test getting an element at a negative index
  2. create another test method to test getting an element at an index that is too large
Note that it is not necessary to test getting from a valid index if you already tested adding elements to a collection as described above.

Testing Methods that Remove an Element from a Collection by Index

  1. add several elements to the collection
  2. remove an element from the middle of the collection
  3. check the size of the collection
  4. check the element that is now at the index you removed from (check enough fields to be sure it is the one you expect)
  5. in separate test methods:
    1. test removing an element at a negative index
    2. test removing an element at an index that is too large
    3. in these cases, just check that the size of the collection was not changed

Testing Methods that Modify Elements of a Collection

  1. modify the elements (call the method)
  2. check at least one modified element
  3. check at least one unmodified element (if any)
  4. in each case, check the field(s) that could have been modified

Testing Methods that Compute An Aggregate Result over a Collection

This rule applies to methods that find the average of some field of the collection's elements, count the number of elements of the collection that satisfy some property, ...
  1. in one test method, add several elements to the collection, then call the "aggregate" method and check that the result is correct
  2. in another test method, use an empty collection to call the "aggregate" method and check that the result is correct

Testing Methods that Search a Collection

  1. use one test method to test searching an empty collection
  2. use another test method to test searching a collection that does not contain the searched-for object
  3. use a third test method to test searching a collection that does contain the searched-for object

Testing Methods that Return Collections

  1. use one test method to check that the empty collection is returned correctly
  2. in another test method, check that a non-empty collection is returned correctly:
    1. check the size of the collection
    2. check the first and last elements (check enough fields to be sure they are the ones you expect)

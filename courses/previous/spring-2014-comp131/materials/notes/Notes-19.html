Because the checking and savings fields are Accounts, it is possible to use the methods already defined in the Account class to do work of depositing and withdrawing for us.

 

For example, in depositChecking we will want to invoke the deposit method on the Account object referred to by the checking field.  If we recall, invoking the deposit method causes the balance to be adjusted appropriately.

 

To invoke a method on a specific object:

            In BlueJ we right click on the object we want to invoke the method on.

                        e.g. on the check or save object.

 

What we need is a way to do the same thing automatically in the code for depositChecking or withdrawSavings etc...

 

To invoke a method on an object we us what is called Dot Notation:

 

            <object reference>.<method name>(<actual parameters>);

e.g. checking.deposit(100);

                       

Another important term is:

Calling Object

The calling object is the object on which the method should operate. I.e. which account do we want to deposit to?

 

In code we give the name of the calling object followed by a dot to tell the computer which object to use.

 

When working manually in BlueJ, we point and right click on the object to tell the computer which object should be used as the calling object.

 

So let’s look at how we would write the depositChecking method:

 

    /**

     * deposit into checking account

     * @param depCheckAmount the amount to deposit

     */

    public void depositChecking(int depCheckAmount) {

        checking.deposit(depCheckAmount);

    }

 

The line in the body of this method invokes the deposit method on the checking object and passes the amount to be deposited as the parameter.

 

Essentially this line asks the Account object referred to by the reference in the checking field to perform the deposit for us.

Creating all of the little objects needed for testing with composition can become seriously tedious...    ----          Test Fixtures

 

A test fixture is a collection of objects that are automatically recreated before each test method is run.

Sometimes you may discover later that you need to add more objects to your test fixture. To do this you need to be a little careful:

 

            Adding to a Test Fixture:

                        Move “Test Fixture to Object Bench”

                        Add new objects

                        Move “Object Bench to Test Fixture”

Failure to do it this way will lead to much pain and suffering!

 

So we’ve seen how to use mutators with composition.  Let’s take a look at using accessors with composition now.  Recall that accessors return values when they are called.  We will often want to do something with the value that is returned. 

 

For example, consider a method to compute a customer’s total balance. To implement this method we’ll want something like:

 

            /**

             * Get the total balance of this customer’s checking and

             * savings accounts.

             * @return the total balance.

             */

            public int getTotalBalance() {

                        // get the checking balance

                        // get the savings balance

                        // add them together

                        // return the result

}          

            To do this we can use a local variable:

                        int checkBal = checking.getBalance();

                       

            What would we write to get the savings balance?

                        int saveBal = savings.getBalance();

 

            And add them together?

                        int totalBal = checkBal + saveBal;

 

            Then return the result:

                        return totalBal;

 

Code-a-little, Test-a-little.

 

Now note that we could also write this without the use of local variables. 

            return checking.getBalance() + savings.getBalance();

 

How do I know if this new and improved version works?  Re-run the test!

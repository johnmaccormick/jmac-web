 Let’s add:   High Class Cohesion

 

The idea is that each class should represent one well-defined entity in the application domain.

 

For example: The Room class is highly cohesive.

It represents everything having to do with a location in the game and nothing more.

 

Now let’s consider the task of adding items to the game.

         We’ll assume that each item has a name and a description.

                  E.g.    book

                           An old dusty book of magical incantations.

 

For now assume that we want to:

Allow one item in each room.

         Allow the player to carry one item.

 

We have essentially two implementation options here:

Add an itemName and itemDescription field to the Room and Player classes.

 

Create an Item class and add a field of type Item to the Room and Player classes.

 

It should be fairly clear that the second option is more cohesive than the first.

With the first option the Room class is now representing the item and the location.

 

With the second option the Room is still just a location and the Item represents everything there is about an item and nothing more.

 

Benefits:

Readability / Maintainability: it is much easier to figure out where the code related to items might be if there is an Item class in the application than if not.

 

Reusability: The Item class could be reused in any application that needs to represent items (e.g. an Antique store inventory program).  If we roll that into the Room class then we can’t reuse the Item functionality without also taking along the Room functionality.

 

Extensibility: The second design will be much easier to extend when we want to... say allow each Room as well as the Player to hold multiple (i.e. a collection) of items.

 

Now imagine that we would like to allow multiple Item objects in each Room and allow the Player to carry multiple Item objects.

So when the player enters the room the game might display a list of the names of the items that are in the room. The player could then pickup an item by name or drop an item.  The player could also request the description of an item by looking at it.

 

In order to implement this functionality we have essentially 3 choices:

         Add multiple Item fields to the Room and Player.

         Add an ArrayList or HashMap of Items to the Room and Player.

         Create an ItemCollection class and use it in both Room and Player.

 

The first two options are not cohesive.  In each case the Room is trying to represent both a location and a collection of items. 

 

The first approach will not be very extensible because each time we want to allow more items in the Room or Player we’ll have to go back and update the code.

 

Both the first and second approaches will result in us having to repeat a lot of code in both Room and Player to manage the Item objects.

 

The best solution is to create the ItemCollection class that represents a collection of Item objects and then use that in both the Room and the Player.  This class can contain the code for adding and removing items from the collection, getting the description of a specific item, getting a list of all of the item names, etc...

         All of the functionality of that will be needed in both Room and Player.

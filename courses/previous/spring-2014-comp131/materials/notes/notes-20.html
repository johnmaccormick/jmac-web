What is a Calling Object?

 

What is the significance of the Calling Object?

 

Listing the calling object in code is the equivalent of pointing to the object in BlueJ.  It tells the code in the methods being called, which object to operate on.

 

When the method is executed it uses the fields of the calling object.  E.g. When deposit is called on checking, the code in the deposit method uses the fields in the checking object. 

 

Now lets look at a slightly different example.  In those examples we were invoking a method on a different object from the one doing the invoking. 

 

In BankCustomer we invoked the deposit method on the Account object referred to by checking.

 

Today we are going to look at how to invoke a method using the same calling object.

 

         For example: Imagine we want to transfer money from savings to checking.

 

To do this we can create a method in the BankCustomer class that when invoked will in turn call the withdrawSavings method on that same BankCustomer to take money out of the savings and then call the depositChecking method on that same BankCustomer to put the money into the checking.

 

Look at savingsToChecking method:

 

/**

 * Transfer funds from savings account to

 * checking account.

 * @param transSCAmount amount the amount to transfer.

 */

public void savingsToChecking(int transSCAmount) {

     this.withdrawSavings(transSCAmount);

     this.depositChecking(transSCAmount);

}

 

Here the calling object is identified by the Java keyword this.

In java this is always a reference to the calling object. 

 

Therefore the withdrawSavings and depositChecking methods will be called using the same object that was used to call the savingsToChecking method, which is exactly what we want.

 

Let’s look at the object diagram.

 

When executed what this method will do is:

Transfer execution to the withdrawSavings method passing transSCAmount

 

                  The withdrawSavings method executes doing its thing.

                  When it is finished execution returns to where it left off.

 

Execution then transfers to the depositChecking method passing transSCAmount.

                  The depositChecking method executes, doing its thing.

 

Trace it with the debugger.

When we step into withdraw where does the this reference point?

What about when we step into deposit?

 

The this reference always points to the object that is currently being operated on – i.e. the calling object.

The this reference also has another use in Java:

 

         It can be used when two variables have the same name.

                  E.g. a field and a formal parameter.

 

         Where this usually shows up is in creating a constructor for a class.

        

Use the BankCustomer as an example:

 

We can either use different names – which is what we’ve done so far or to use the keyword this.

 

The keyword this provides a way to refer to the field:

         this.name = name;

 

The use of this indicates that the name on the left hand side of the assignment the name variable associated with the calling object (i.e. the field). The name on the right side is the parameter named name.

 

 

For this class you are free to use parameters with the same name as fields in your constructors if you like. However, because it is a questionable programming practice in other locations, WebCAT will deduct points for doing this in any method other than the constructor.

The line of code invokes the deposit method on the Account object referred to by savings.

But there is no such object – the reference is null!!

 

                           What happens?

                           The result is something called a

NullPointerException

                           The calling object has the value null!

 

When a null pointer exception occurs two things happen:

Error message indicating the file and line number where the error occurred.

 

Source code window opens, highlighting the line where the error occurred.

 

In general if you see this error, the problem is that the calling object has the value null.

                  Most often you have forgotten to initialize a field in the constructor.

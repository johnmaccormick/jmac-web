Topic #37: Zuul – Responsibility Driven Design

 

Last time we discussed:

 

Designing Good Software:

High Method Cohesion

Loose Coupling

 

What we are doing in talking about these things and in working with zuul is starting to learn about how to design good software and what happens when the design is bad.

 

            For example:

Zuul started with low method cohesion and tightly coupled classes.

 

The low method cohesion led to repeated code, which was problematic because changes have to be made everywhere the code is repeated.

 

The use of public fields created tight coupling between the classes.  This lead to rippling changes.  That is, a change in one place resulted in the need to make lots of changes in other places.  In other words, it makes refactoring a class much more difficult because changes to the class necessitate changes elsewhere.

 

Making the fields in the Room class private resulted in the need to change a number of things in both the Game class and the Room class.

 

Whereas, once we made those fields private, we could actually eliminate them completely in favor of a HashMap without making any changes anywhere else in the code.

 

 

The idea of loose coupling is closely related to what is called:

 

Encapsulation

 

The idea is that a Class should encapsulate its data.

It should not be possible to directly manipulate the data from outside the class.

 

If a class encapsulates its data, then other classes cannot be tightly coupled to the class through its data.

 

 

 

 

 

 

 

 

 

Responsibility Driven Design

 

Each class should have a well-defined role in the application.

Each class should be responsible for handling its own data.

 

When we started zuul – each class did have a well-defined role.

            Command is responsible for representing user commands.

            CommandReader is responsible for reading commands.

            Player is responsible for keeping track of the player.

            Room is responsible for representing world locations.

            Game is responsible for the overall game play.

 

However, each class was not responsible for handling its own data:

Generating a list of the exits from a Room was done by the Game class. But the data about the exits is defined in the Room class (as it should be) so the Room class should really be responsible for generating the list of exits.  You fixed this in the lab by adding the getExitString method to the Room class.

 

Similarly, the Game class was generating the description of the current room and displaying it.  But again this is something dealing with the Room’s data so the Room should be responsible for generating this description.  You fixed this in the lab by adding the getLongDescription method to the Room class.

 

Now each Room is responsible for generating its own description.  When the Game wants to print the description of a Room it asks the Room for a description of itself and then prints it.

 

The idea of responsibility driven design is very useful for:

 

            Selecting the classes that make up a new application.

            Deciding where to add new functionality to an application.

 

E.g. If we want to allow each Room to contain an item, then the Room class should be responsible for keeping track of the item that is in the room.  Further, when the Room generates its description it will also include information about the item in the description.

 

Similarly, if we want to allow the Player to pick up an item, then the Player class should be responsible for keeping track of the item that the Player is carrying.

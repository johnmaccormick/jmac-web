Composition:

         This is a technique for assembling objects into larger software system.

 

Up to this point we’ve been writing individual classes and using them to create objects.

 

As we begin to construct more complex and involved software systems we need techniques that make the added complexity manageable.  Composition is one such technique.

 

Composition allows us to build complex software in much the same way that we build complex things in the real world.  We build them from many smaller simpler parts. 

 

A car does not pop fully formed into the world.  Instead it is assembled using many smaller parts that may also each be assembled using many smaller parts.

 

If we think of Tire, Door, Radio, Engine and Transmission as classes, then a car is composed of objects of these types.  4 Tire objects, two Door objects, one Radio object, one Engine object and one Transmission object.  These are components of a car.

 

In our code we will be implementing composition by having fields in our classes that have their data type defined by other classes.

 

We have some experience using a system like this.  In lab 3 you created DiseaseSimulators by using a Disease object and a Population object.  That is the DiseaseSimulator was composed of a Disease and a Population.

 

 

Let’s look at an example where this type of Composition might be useful:

 

Consider a system where we need to keep information about Students – maybe something for a college Registrar’s office.

 

We might need to keep track of things such as the student’s name, id number, home address and the courses that are being taken.  Probably a bunch of other stuff too... but we’ll keep it simple for now.

 

Given what we’ve done so far in this class, we might attack the problem as  in the BadStudentDesign project.

 

         One field for each piece of information about the student.

 

Then we would need to add constructors, accessors and mutators that appropriately, initialize, return and change this information.  E.g. adding a course, dropping a course, changing the student’s home address if his/her parent’s move.

 

This seems like a tedious and cumbersome approach to the problem.

 

What we might recognize is that some collections of these fields actually represent separate objects themselves.

        

The address information for home and local address both represent addresses.  So we should be able to make a class for representing Addresses and use it.

 

The information about each course represents a course.  So we should be able to create a class for representing Courses and use it.

 

So let’s look at a better design that uses the idea of Composition:

 

         The GoodStudentDesign project has three classes:

                  Student

                  Address

                  Course

 

         Look at the Student class:

We would say that a Student object is composed of two Address objects and four Course objects.

 

Recall that we said earlier in the semester that every class defines a data type.  So we are able to use Address and Course as the types of fields just like int, String and double.

 

In fact, String is actually a class provided with Java...

 

We can also use them as the data type of local variables, formal parameters and method return types.  So we could have a method named getAddress that would have a return type of Address and would return an object of type Address.

 

         We’ll see some of this over the next few classes.

 

In building our software this way we are making use of two:

         Big Ideas in Software Development:

                  Divide and Conquer

                  Modularization

 

Divide and Conquer: Breaking large, complex problems down into smaller, simpler sub-problems.

Continue breaking down the sub-problems until they are manageable.

                  (for example, a solution can be implemented as a single Java class)

Benefit:

It is easier to solve smaller problems.

It is easier to build a radio than it is a car.

It is easier to build an Address class and a Course class than it is one big Student class.

 

Modularization: problem is divided so that sub-problem solutions:

are relatively independent

interact in well-defined ways

 

Benefits:

Division of labor

 

With software, modularization makes it possible to have different programmers solve different sub-problems. For example, one programmer could write the Address class, one the Course class and another the Student class.  Once those pieces have been created and tested they can be brought back together to produce the whole program.

 

In really large projects involving 1000’s of classes and millions of lines of code this division of labor is necessary.

 

Changes are localized to a small number of classes

If we decide to change the way the radio works (CD player vs. cassette), it doesn’t affect the way the rest of the car is built. This is true because the car and the radio interact in a well defined way and all the different implementations of radios interact in the same way.

 

Similarly, if we decide to change the way the Course class works (e.g. by including an Instructor object instead of a String) it will probably not affect the way the Student class works.

This design highlights another two:

Big Ideas in Software Development:

                  Divide and Conquer

                  Modularization

Reuse

Abstraction

 

Reuse

         Reusing existing components in new software.

         We don’t reinvent the wheel every time we want to build a car.

Same thing for software...

Once we have created and tested an Address class and an Account class we don’t have to write or test either of those when creating our BankCustomer class.  We can just make use of them.

 

Similarly, if sometime in the future we want to extend the Student class to keep track of say a decliningBalance account, we could reuse the Account class again, saving significant time and effort because we don’t have to write or test the code.

 

 

Abstraction

Using a component without worrying about its implementation details.

 

Once a radio has been produced, a car designer doesn’t need to understand how it works in order to incorporate it into a car. The designer only needs to know how to interact with the radio and what it does.  What size space is needed, what incoming and outgoing electrical connections are needed etc...

 

Similarly, once a programmer has created and tested the Account class we don’t have to know about the implementation details of that class in order to use it in a BankCustomer class. It is not necessary to worry about how each of the methods in the class works, we just need to know how to interact with it.  E.g. that it has a deposit method that we call when we want to add money to the account and a withdraw method that we call to remove money from the account.  From the point of view of using the Account class we don’t need to concern ourselves with what fields it has or how it does the deposit or withdraw.

 

All of these big ideas are really tools for managing complexity.

Humans do not deal well with massive amounts of detail.

It is impossible for us to think about every detail of a large software system.

These big ideas allow us to focus on one small part of the problem at a time.

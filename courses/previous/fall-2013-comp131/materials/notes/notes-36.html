Code Duplication and Method Cohesion

 

Note that code for printing information about the player's location is repeated in both printWelcome and goRoom

 

Why is this duplication bad? 

If the code needs to be changed, the programmer very well might forget to modify both/all places.

 

This type of code duplication is related to what is called Method Cohesion:

Each method should be responsible for one well-defined task.

         Makes methods easier to write

         Makes code easier to understand and easier to maintain.

 

         You should:

         Strive for high method cohesion

         Avoid low method cohesion

 

printWelcome has low method cohesion because it prints the welcome information and the location information.

 

goRoom has low method cohesion because it moves the player to a new location and prints the location information.

 

Solution?     Add a new method called printLocationInfo that prints the location information.

Coupling:

 

Note that the Game and Player classes use code like:

currentRoom.northExit 

in many, many places (show the goRoom method in Player)

 

This is a direct reference to a field of the Room class.

 

It is possible because the fields in Room were declared public.

We’ve said not to do this, which is the right choice almost 100% of the time. Now we’ll take a look at why that is true.

 

Using these fields directly is bad design!       Why?

 

What if we change the implementation of the Room class by doing something as simple as changing the name of one of the fields?

 

We will have to change code in many places in the Game and Player classes.   Though the compiler will tell us each place that needs to be changed.

 

What about another kind of change, suppose we wanted to add another kind of exit (up, elevator, slide, ...)?             What is involved now?

 

This type of rippling change results from what is called Coupling:

The Game and Player classes are tightly coupled with the Room class.

 

Tight coupling arises when classes do not communicate via a well-defined interface, i.e. through accessor and mutator methods instead of via direct use of fields.

 

Any changes to the Room class will force changes to the Game and Player classes.

 

                  When designing programs you should:

         Strive for loose coupling.       Avoid tight coupling.

 

Loose Coupling:

         Better abstraction

         Higher reuse

It is possible to read and understand one without understanding all details of the other

                  Easier updates and maintenance

It is possible to change one without necessarily changing the other

                           Locality of change

 

So we should refactor the Game, Player and Room classes to reduce their degree of coupling. 

 

Approach:

Declare the northExit, southExit, ... fields of Room as private

 

Add a getExit method to the Room class that takes a direction, and returns the Room in that direction (or null if there is no such room).

 

    /**

     * Return the room that lies in the indicated direction, or

     * null if there

     * is no exit in that direction.

     * @param direction the direction of the exit

     * @return the Room that lies in that direction

     */

    public Room getExit(String direction) {

        if (direction.equals("north")) {

            return northExit;

        } else if (direction.equals("east")) {

            return eastExit;

        } else if (direction.equals("south")) {

            return southExit;

        } else if (direction.equals("west")) {

            return westExit;

        } else {

            return null;

        }

     }

 

 

Note that this will also allow us to more easily have exits in directions other than the standard compass directions.  To get the exit in a particular direction we would need to add a field and then add an else if clause for the direction here.

 

We would also need a way to set the Room that is in this new direction.

         Could just add another parameter to the setExits method.

         That seems tedious because we could imagine having lots of exit types.

Some of which are used very rarely and it would seem silly to have to specify all of them all of the time.

 

Modify the setExits method as follows:

Change the name to setExit (it will now set only one exit at a time)

Have it take a direction and a Room as parameters

Sets the indicated exit to the indicated Room

 

This will allow us to more easily have exits other than the standard compass directions.

 

    /**

     * either leads to another room or is null (no exit there).

     * @param direction the direction of the exit

     * @param exit the Room that lies in that direction

     */

    public void setExit(String direction, Room exit)

With these changes, outside code cannot access the fields of the Room class directly.

         A class with this property is said to encapsulate its own data.

 

Encapsulation is the primary way of implementing information hiding.

Encapsulation, information hiding and loose coupling are all closely related.

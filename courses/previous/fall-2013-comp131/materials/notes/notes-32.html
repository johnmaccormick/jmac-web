A Problem with this code:

We have repeated code for finding a particular class in the schedule at two different places in the code:

In isTaking we iterate through the schedule looking for a course.

In dropCourse we also iterate through the schedule looking for a course.

 

In both places we go through the courses one by one looking for a specific course identified by department and number.

The code may not look exactly the same in both places but it does in fact do the same thing in both places.

 

Is this repetition a problem?

 

What happens if we change the way that courses are identified?

e.g. add a courseID field, or identify courses by department, number, semester and year.

 

It would be very easy forget to update all of the places were courses are found by department and number.  It would be much better to update it in only a single place!!!  Repeated code is (almost) always bad

 

Solution: Put the common code into a method that is called by both isTaking and dropCourse.

 

Lets improve our Student class by writing a new method that returns the index of a course in the student's schedule, or -1 if the course is not in the schedule.

 

Copy and paste the code from dropCouse and then change the body of the trailing if/else statements.

 

     /**

      * return the index of a course in a student's schedule, or -1 if the

      * course does not occur

      * @param department the department of the course

      * @param number the number of the course

      * @return the index of a course in a student's schedule, or -1 if the

      * course does not occur

      */

     public int findCoursePosition(String department, int number) {

         int index = 0;

        boolean found = false;

         while (!found && index < schedule.size()) {

             Course current = schedule.get(index);

             if (department.equals(current.getDepartment()) &&

                 number == current.getNumber()) {

                 found = true;

             } else {

                 index++;

             }

         }

         if (found) {

             return index;

         } else {

             return -1;

         }

     }

Having the tests lets us change the code with confidence that we have not broken anything.

 

What we’ve just done is called refactoring.

 

Refactoring - improving the implementation of a class without changing its public interface.

 

Public Interface - the methods that are available to other classes.

Includes name, parameter list, return type, behavior

 

If we change the implementation of a class without changing its public interface that means that any other code that uses the class will continue to work without modification.

 

Inevitably, some refactorings will change the public interface to reflect new features or changes in organizational policy, but this should be avoided when possible

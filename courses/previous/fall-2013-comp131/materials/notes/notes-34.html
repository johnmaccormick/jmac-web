Last time we ran into a problem with changing “sophomores” to “second years.”

The problem was that we changed the term in one place and then “forgot” to change it in another.

 

Now we could just fix our code in the other place, but this problem comes up a lot. 

         It came up before with repeated code.

         It comes up here with constant values being repeated at multiple places.

You can imagine this one being pretty bad in a real registrar’s database system. 

 

It would be much better if we could define these types of values in just one place.

It would be easier to find

We would only need to make one change to adjust for changing college policies

It would be a bonus if this definition could not be changed by accident

 

Java (and most other languages as well) provides a means for defining constants which are exactly what we want.

 

Example:

                  public static final String FOURTH_YEAR = "senior";

 

Now anytime we want to talk about fourth year students we can use Student.FOURTH_YEAR to refer to this value ("senior") anywhere in the program. Further, if the term “senior” becomes politically incorrect and we change the value assigned to the constant, to say “supreme ruler” instead of “senior” then any code that uses the constant will still work!

 

For example:

Declare the 5 constants in Student:

         public static final String FOURTH_YEAR = "senior";

     public static final String THIRD_YEAR = "junior";

     public static final String SECOND_YEAR = "sophomore";

     public static final String FIRST_YEAR = "first year";

     public static final String ALUMNUS = "potential donor";

 

Replace the standings in Student and printStandingStats

Note two things:

Constants declared public can be used in other classes.

We use the name of the class when referencing constants.

 

So let’s take a closer look at the constant declaration now – in particular the new Java keywords that appear in the definition.

 

         static:

         All instances of the class share one copy of the field

Recall that if a field is not static every instance of a class has its own copy of the field.

 

This is helpful because there is no need to waste memory by having every instance of the Student class store possible standings

 

A static field belongs to the class, not to instances of the class

Because of this a static field is referred to using the name of the class (not the name of an instance).

 

e.g. Student.FIRST_YEAR     not current.FIRST_YEAR

final:

The field can NOT be modified after being initialized

This is helpful because we do not want any code to change the standings.

static final:

Called a constant or class constant because it belongs to the class and it cannot be changed

 

Writing constants in all capital letters (with underscores between words) is a common Java style (a Java convention) and will be checked by WebCAT.

 

Constants of any Java type can be used - the type is specified in the constant declaration

 

 

Hash Maps:

 

Potential problem with our solution so far:

Each time a student adds or drops a course, we must search the list of all students

If there are hundreds or thousands of students, this could take a long time

 

It would be great if we had a more efficient way to find a Student object based on the student ID

The solution is something called a HashMap. 

A  HashMap,  given a key value like a student ID, can find the Student with that ID without looking through all of the Students.

We won’t see how it works here, we are just going to use abstraction to allow us to make use of a HashMap to improve our RegistrarsOffice class. If you end up really wanting to know how it works – you can continue on to CS132 and then 232 where we talk about how to implement HashMaps.

 

The HashMap Class

 

The HashMap class is located in package java.util (just like ArrayList)

So in any class that uses a HashMap, we will have to put the line:

                  import java.util.HashMap;

         at the beginning of the class definition (again, just like ArrayList).

 

A HashMap object holds a collection of key and value pairs

A key is used to "look up" the value.

 

For example:

In a telephone book, a name is the key and their telephone number is the value.

                  In a dictionary a word would be the key and the definition the value.

 

We can draw a 2 column table with key / value as the columns.

         Put in names and phone numbers.

 

The thing about a HashMap is that when given a key, it doesn’t have to scan through the key’s one by one.  It is able to just jump right to the correct spot for the key and get the associated value.

 

If we have the key it is much more efficient to find the value in a HashMap that it would be to find it in an ArrayList by simply starting at the beginning and checking each entry in the list one-by-one.

 

The most important methods in class HashMap are:

 

put: takes a key and value and adds them to the HashMap

get: takes a key and returns the associated value (or null)

 

The get operation is very efficient - much more so than searching an ArrayList. To find a value in an ArrayList we’d look at about 1/2 of the objects in the ArrayList on average.  The amazing thing is that a HashMap will have to look at only 1 or 2 objects on average before finding the value.

         If you find that amazing and wonder how... take a few more CS classes!

 

See FirstHashMap project for an example of using put and get

 

         One field of type HashMap.

Like arrays and ArrayLists, the constructor must initialize the field by creating a new HashMap object.  The HashMap constructor does not take any parameters.

If there is no value associated with the key used in a call to get, the get method returns null.

Comment on "commenting out"

This is typical of what professional programmers might do during a refactoring - leave the old version of the code in the class as a comment so that it is still available in case the refactoring doesn't work.  The programmer would then go back and delete the old code once the refactoring has been completed and tested.

 

Note that in the homework for today we are asking you to comment out the old versions of the method bodies, instead of deleting them, so that we can grade both the original version and the refactored one.
